# mdmiel 要件定義書

- ステータス: 実装反映済み ( フェーズ2.6まで )。フェーズ3以降は計画
- 最終更新: 2026-07-14

## 背景・課題

AIエージェント ( Claude等 ) に要件定義書をMarkdownで、画面モックをHTMLで、さらに実装コードまで作らせる開発フローが定着しつつある。しかし成果物のレビュー手段が追いついていない。

- 要件の文章と画面イメージが別ファイルで、対応関係を見比べられない
- 「この段落のここが違う」という行単位のフィードバックを残す場所が無い
- フィードバックを人が口頭・チャットで伝えると、AIに機械的に読み戻せず修正ループが回らない
- AIが書いたコードを人が全量読んでレビューするのは認知負荷が高すぎ、AI駆動開発のボトルネックになる

## 目的

Markdown/HTMLをローカルサーバーで並べて閲覧し、行単位の付箋コメントを貼れるレビューツールを提供する。コメントは機械可読な形式でファイルに残し、AIが読んで成果物を修正するループを成立させる。

位置づけは「AI成果物のレビュー基地」。閲覧機能そのものの豊かさ ( シンタックスハイライト等 ) は追わず、共有・コメント・md↔html紐づけを本線とする。

狙う効果はAI駆動開発時代のレビュー負荷の軽減。人が生のコードや成果物を個別に全量読むのではなく、AIにmd化させた成果物同士を突き合わせて差分・対応関係をレビューするスタイルへ移行させる。

## 想定ワークフロー

要件・モックのレビュー ( 実装前 ):

1. Claudeが要件定義mdを作成する
2. Claude ( design ) がそれを読んで画面モックhtmlを作成する
3. 人がmdmielで両者を並べて閲覧し、行単位の付箋コメントでフィードバックする ( ブラウザだけで完結するため非エンジニアも参加できる )
4. コメントはサイドカーJSONに保存されるため、Claudeがそれを読んで要件・モックを修正する
5. 2〜4を繰り返す

実装の理解・レビュー ( 実装後 ):

6. Claudeが実装コードから解説ドキュメントmd ( 設計書。コード抜粋込み ) を生成する
7. 人は「要件定義md×解説md」を並べて要件が実装に反映されているかをレビューし、「解説md×コード抜粋」で実装の妥当性をレビューする。コードを全量読む代わりに、md化された解説を入口にすることで認知負荷を下げる
8. 疑問・指摘は解説mdの該当行に付箋コメントで残し、Claudeが回答・修正・解説の改訂を行う

この実践例が docs/codetour/ ( コードから生成した解説ドキュメントをmdmiel自身でレビューする構成 )。

## 利用者

- レビューする人: 開発者・非エンジニアの関係者。ブラウザ操作のみ
- レビューされる側: AIエージェント。サイドカーJSONをファイルとして読み書き
- 運用する人: 開発者。CLIひとつで起動できること

## 機能要件

| # | 要件 | 状態 |
|---|---|---|
| F1 | 指定ディレクトリ配下のmd/htmlファイルをツリー一覧表示する | 実装済み |
| F2 | Markdownをレンダリング表示する ( 生HTMLはエスケープ ) | 実装済み |
| F3 | HTMLモックをサンドボックスiframeで表示する ( 相対パスのCSS/画像も解決 ) | 実装済み |
| F4 | 左右2ペインで任意の2ファイルを並べて表示する ( split ) | 実装済み |
| F5 | 共有用リンクURLを発行できる。mdは行リンク ( 該当行へスクロール+ハイライト )、htmlは付箋リンク ( /#/comment/<id>、開くと該当付箋へスクロール+フラッシュ ) | 実装済み |
| F6 | 行単位でコメントを追加・編集・削除・解決 ( resolved ) できる | 実装済み |
| F7 | コメントは対象行に貼り付く付箋カードとして常時表示する ( md/html両ペイン ) | 実装済み |
| F8 | 付箋はドラッグで移動でき、位置は保存され再訪時も再現される | 実装済み |
| F9 | コメントは1件=1ファイルのサイドカーJSONに保存する ( Git共有でコンフリクトしない ) | 実装済み |
| F10 | ファイル更新で行がずれてもコメントを正しい行に追従させる ( 不能ならorphaned表示 ) | 実装済み |
| F11 | ファイル変更を検知してプレビューを自動リロードする | フェーズ3計画 |
| F12 | シンタックスハイライト・Mermaid・数式などのmdリッチ表示 | フェーズ3計画 ( 優先度低 ) |

## 画面要件

モック作成・UI変更時はこの構成を前提とする。

- ヘッダ: プロダクト名、コメントパネルの表示切替ボタン
- サイドバー ( 左 ): md/htmlファイルのツリー。クリックで左ペインに表示、Shift+クリックまたは専用ボタンで右ペインに表示
- コンテンツ領域: 1ペインまたは左右split。各ペインは独立スクロール
- 行アクション: mdでは行の右クリックでメニューを固定表示し、「リンクをコピー」「コメント追加」を提供する ( ホバー起動は採用しない )。htmlの行リンクは廃止し、付箋カードの「リンクをコピー」で付箋リンクを発行する
- 付箋カード: 対象行の横に重なって表示。折りたたみ時は本文抜粋+作者+バッジ ( resolved )、クリックで展開して全文・編集・削除・解決切替。ヘッダをドラッグで移動
- 未解決セクション: 行に貼れなくなった ( orphaned ) 付箋はコメント一覧パネル下部にまとめて表示し、ヘッダーのコメントボタンに未解決件数を表示する ( ペイン上には出さない )
- コメントパネル ( 補助 ): コメントの一覧。既定は閉じる

## 非機能要件

- 配布: 単一バイナリ。Go製CLIにフロントエンドを埋め込み、`mdmiel <dir>` だけで起動しブラウザを自動で開く。サブコマンドは設けない ( 人向けの閲覧ツールであり、機能追加はCLIでなくweb UI側に足す方針 )
- 対応OS: macOS / Windows / Linux
- セキュリティ: ローカル専用とする
  - 127.0.0.1のみにバインドし、LANに公開しない
  - パストラバーサルを拒否する ( 対象ディレクトリ外・ドットファイルへのアクセスは403 )
  - DNSリバインディング対策としてHostヘッダを検証する
  - CSRF対策として状態変更APIのOriginヘッダを検証する
- 依存: バックエンドはGo標準ライブラリのみで実装する ( 供給網リスクとメンテコストの最小化 )
- データの所在: コメントは対象ディレクトリ配下 `.mdmiel/comments/<id>.json`。ツール本体を消してもデータはプレーンなJSONとして残る

## スコープ外

- ファイル本文の編集 ( 編集はVSCode等の既存エディタで行う。mdmielはビューア+コメントに徹する )
- サーバー/DBでのコメント共有 ( 将来のチーム対応で検討。ストレージ層は差し替え可能に作ってある )
- md/html以外のファイル形式の表示
- 認証・マルチユーザー管理 ( ローカル専用のため )

## 受け入れ基準 ( 抜粋 )

- `mdmiel <dir>` で起動し、ブラウザでmd/htmlの一覧・レンダリング・splitが動く
- 行コメントを追加すると `.mdmiel/comments/<id>.json` が生成され、リロード後も同じ行に付箋が表示される
- コメントを1件ずつ追加した2つのGitブランチをマージしてもコンフリクトしない
- 対象ファイルに行を追加してもコメントが正しい行に追従する
- `path=../../etc/hosts` のようなトラバーサル要求、LANからのアクセス、異Originの状態変更要求がいずれも拒否される
